# Loss Scan Quick Start

AIに仕事を任せる前に、失われやすい価値を短時間で確認するための実践ガイドです。

> Version: v0.1  
> Estimated time: 30秒〜3分

## Loss Scanとは

AIは、与えられた指示と参照可能な文脈から、もっともらしい成果物を高速に生成します。

しかし、その仕事に固有の業務ルール、過去の障害、既存設計の理由、運用上の制約まで、自動的に理解しているとは限りません。

Loss Scanは、AIへ作業を委譲する直前に、失われる可能性があるものを3つの質問で確認する方法です。

目的は、すべてのリスクを洗い出すことではありません。

**最も壊れやすい1〜3点を先に見つけ、AIへの指示、レビュー、テスト、監視に反映すること**が目的です。

---

## 3つの質問

### 1. AIは、この仕事の何を知らないか？

AIに渡していない固有の文脈を確認します。

例：

- 業務ルール
- 顧客固有の事情
- 過去の障害
- 既存設計を選んだ理由
- 運用上の制約
- 暗黙の例外処理
- 社内規程
- データの意味や定義

### 2. この仕事で、一番壊れやすい場所はどこか？

失敗した場合の影響が大きい境界を確認します。

例：

- 認証・認可
- 個人情報・機密情報
- 決済・課金
- 外部API
- DB migration
- リトライ・二重実行
- 並行処理
- タイムアウト
- 境界値
- 既存システムとの接続
- 人間とAIの責任境界

### 3. 壊れたとき、どうやって気づくか？

失敗を検知・追跡できる状態かを確認します。

例：

- 自動テスト
- 構造化ログ
- メトリクス
- トレース
- アラート
- 監査ログ
- ダッシュボード
- ユーザーからの問い合わせ
- 人間によるレビュー
- ロールバック手順

## 30秒版

低リスクな作業では、次の3行だけを埋めます。

```text
AIが知らない文脈：
一番壊れやすい場所：
壊れたときの検知方法：
```

少なくとも一つ書けたら、それをAIへの指示やレビュー観点に追加します。

---

## AIへの依頼テンプレート

```text
以下の作業を進める前に、Loss Scanを行ってください。

1. あなたが知らない可能性がある業務・設計・運用上の文脈
2. 最も壊れやすい場所
3. 壊れた場合の検知方法

重要度の高いものを最大3点に絞ってください。

その後、これらを落とさない実装案を提示し、
未確認の前提と残存リスクを明示してください。
```

AI自身にLoss Scanを出力させる場合も、最終的な判断は人間が行います。

## 記入例：外部メール送信を伴うユーザー登録

### 作業

AIに、ユーザー登録APIと確認メール送信処理の実装を依頼する。

### Loss Scan

```text
AIが知らない文脈：
同じメールアドレスの再登録ルールと、
メール送信サービス障害時の既存運用を知らない。

一番壊れやすい場所：
登録処理は成功したがメール送信だけ失敗する場合と、
リトライによる二重送信。

壊れたときの検知方法：
送信結果を構造化ログに残し、
requestIdで登録処理とメール送信を追跡する。
失敗率をメトリクス化してアラートを設定する。
```

### AIへの追加指示

```text
ユーザー登録とメール送信を一つの成功として扱わず、
それぞれの状態を追跡できるようにしてください。

メール送信の失敗、タイムアウト、リトライ、
二重送信を考慮してください。

登録処理とメール送信をrequestIdで追跡できる
ログとメトリクスの案も提示してください。
```

## いつ使うか

Loss Scanは、少なくとも次のタイミングで使います。

- AIへ実装を依頼する前
- AIが生成した大きな変更をレビューする前
- 外部APIやデータベースに影響する変更の前
- 本番環境に影響する操作をAI agentへ許可する前
- AI生成物を顧客や意思決定者へ提示する前
- 過去の障害に関係する箇所を変更する前

---

## 省略してよい場合

以下のような低リスクかつ容易に取り消せる作業では、詳細な記録を省略できます。

- 文案のたたき台
- 誤字脱字の修正
- 破棄可能なローカル実験
- 本番データや外部サービスに触れない調査
- 変更範囲が明確な軽微な整形

ただし、「低リスクだと思った理由」は説明できるようにします。

---

## Loss Scanだけでは不十分な場合

次の変更では、3問に答えるだけで完了としてはいけません。

- 認証・認可
- 決済・課金
- 個人情報・機密情報
- DB migration
- 外部APIへの副作用
- 本番データの変更
- 不可逆な操作
- 法令・契約・安全性に関わる判断

これらでは、必要に応じて次を追加します。

- 設計レビュー
- セキュリティレビュー
- 必須テスト
- CI/CD gate
- 権限分離
- 承認者の指定
- ロールバック計画
- 操作と判断の証跡

Loss Scanは、安全を保証する仕組みではありません。

**どの追加統制が必要かを判断する入口**です。

---

## 利用後に記録すること

Loss Scanを実施した後、次のいずれかが起きたら記録します。

- 事前にLossを発見できた
- レビュー指摘が前倒しになった
- 手戻りを防げた
- 3問では発見できなかった
- チェックリスト化して形骸化した
- 作業に対して重すぎた
- 新しい質問が必要になった

成功だけでなく、機能しなかった事例もプレイブックの改善材料にします。
